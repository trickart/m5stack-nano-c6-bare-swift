# Task 2: ESP32-C6 Bootloader & Flash Image Preparation

## ESP32-C6 Boot Sequence

```
Power ON â†’ ROM Bootloader â†’ 2nd Stage Bootloader â†’ Read Partition Table â†’ Application Start
```

1. **ROM Bootloader** (built into chip, not modifiable)
   - Determines boot mode from GPIO9 strap pin
   - Loads the 2nd stage bootloader from the beginning of SPI Flash (offset `0x0`)
2. **2nd Stage Bootloader** (written in Swift, `Sources/Bootloader/`)
   - Disables watchdog timers
   - Reads the partition table at offset `0x8000`
   - Locates the factory app partition (offset `0x10000`)
   - Loads IRAM/DRAM segments from flash to RAM via direct SPI1 register access
   - Configures Flash MMU for IROM/DROM segments
   - Enables instruction cache
   - Jumps to the app entry point
3. **Application** (custom-built)
   - Execution begins from the entry point

## Flash Layout

| Offset | Content | Size | Notes |
|--------|---------|------|-------|
| `0x00000` | bootloader.bin | ~3KB | Built from `Sources/Bootloader/` |
| `0x08000` | partition-table.bin | 3KB | Generated by `Tools/gen-partition-table.swift` |
| `0x10000` | app.bin | Variable | **Custom Swift application** |

### Partition Table Contents

```
nvs      : data, nvs,     0x9000,  0x6000
phy_init : data, phy,     0xf000,  0x1000
factory  : app,  factory, 0x10000, 0x100000 (1MB)
```

### Partition Table Binary Format

Total size: 3072 bytes (0xC00), written at flash offset `0x8000`.

Each entry is 32 bytes:

| Offset | Size | Content |
|--------|------|---------|
| 0x00 | 2 | Magic (`0xAA 0x50`) |
| 0x02 | 1 | Type (0x00=app, 0x01=data) |
| 0x03 | 1 | SubType |
| 0x04 | 4 | Offset (little-endian) |
| 0x08 | 4 | Size (little-endian) |
| 0x0C | 16 | Label (null-padded) |
| 0x1C | 4 | Flags |

After the last entry, an MD5 checksum entry follows:

| Offset | Size | Content |
|--------|------|---------|
| 0x00 | 2 | MD5 magic (`0xEB 0xEB`) |
| 0x02 | 14 | Padding (`0xFF`) |
| 0x10 | 16 | MD5 digest of all preceding partition entries |

The remainder is padded with `0xFF` to fill 3072 bytes. Generated by `Tools/gen-partition-table.swift`.

## Bootloader Architecture

The bootloader is a bare-metal Swift executable compiled to RISC-V, with all code placed in IRAM (since Flash MMU is not yet configured when the bootloader runs).

### Source Files (`Sources/Bootloader/`)

| File | Purpose |
|------|---------|
| `Bootloader.swift` | Main entry point and boot flow |
| `FlashRead.swift` | SPI flash reading via direct SPI1 register access |
| `FlashConfig.swift` | SPI clock configuration and read mode setup |
| `MMU.swift` | Flash MMU page table configuration |
| `Watchdog.swift` | Watchdog timer disable |
| `Delay.swift` | Delay functions using SYSTIMER |
| `RuntimeStubs.swift` | Minimal C runtime stubs |

### Boot Flow

```
clearBSS() â†’ disableWatchdogs() â†’ configureFlashSPI()
â†’ findFactoryApp()       // Read partition table, locate factory app
â†’ readImageHeader()      // Verify magic (0xE9), get segment count & entry point
â†’ loadRAMSegments()      // Copy IRAM/DRAM segments from flash to RAM
â†’ setupFlashMMU()        // Map IROM/DROM segments via MMU page table
â†’ jump to entry point
```

All flash I/O is performed by directly controlling the SPI1 controller registers (`FlashRead.swift`), with no ROM function dependencies. Cache control (disable/enable, bus shutdown) is also implemented in pure Swift via EXTMEM registers (`MMU.swift`).

### MMU Configuration

The ESP32-C6 MMU maps 64KB flash pages to virtual addresses:
- Page size: 64KB (`0x10000`)
- 256 entries total
- Virtual base: `0x42000000` (shared IROM/DROM address space)
- MMU registers: `SPI_MEM_MMU_ITEM_INDEX` (`0x60002380`), `SPI_MEM_MMU_ITEM_CONTENT` (`0x6000237C`)
- Valid bit: bit 9 (`0x200`)

### SPI Flash Read

SPI flash reading is implemented in pure Swift by directly controlling the SPI1 (SPIMEM1) controller registers, without relying on the ROM function `esp_rom_spiflash_read`.

The ESP32-C6 has two SPI memory controllers:
- **SPI0** (0x6000_2000): Cache controller â€” used by hardware for instruction/data cache
- **SPI1** (0x6000_3000): Flash controller â€” used by software for direct flash access

#### SPI1 Register Map

| Register | Address | Purpose |
|----------|---------|---------|
| `CMD` | 0x6000_3000 | Command trigger (bit 18 = USR) |
| `ADDR` | 0x6000_3004 | Flash address |
| `CTRL` | 0x6000_3008 | Read mode flags (DIO, QIO, etc.) |
| `USER` | 0x6000_3018 | Phase enables (command, addr, dummy, miso) |
| `USER1` | 0x6000_301C | Address bit length, dummy cycle count |
| `USER2` | 0x6000_3020 | Command value and bit length |
| `MISO_DLEN` | 0x6000_3028 | Read data bit length |
| `W0`â€“`W15` | 0x6000_3058â€“0x6000_3094 | Data buffer (64 bytes) |

#### DIO Read Mode Configuration

The flash is configured for **DIO (Dual I/O)** mode:

- **USER_REG** (0x6000_3018): USR_COMMAND=1, USR_ADDR=1, USR_DUMMY=1, USR_MISO=1, USR_MOSI=0. Read-modify-write to preserve CS_SETUP/CS_HOLD bits.
- **USER1_REG** (0x6000_301C): USR_ADDR_BITLEN=27 (28-bit: 24-bit address + 4-bit mode for DIO 0xBB command), USR_DUMMY_CYCLELEN=1 (2 dummy cycles).
- **USER2_REG** (0x6000_3020): USR_COMMAND_BITLEN=7 (8-bit), USR_COMMAND_VALUE=0xBB (Fast Read Dual I/O).

#### Read Sequence

Each read transfers up to 16 bytes:

```
1. ADDR_REG = flash_address       (raw value, lower 28 bits used)
2. MISO_DLEN = (byte_count * 8) - 1  (read length in bits, minus 1)
3. CMD_REG |= (1 << 18)           (trigger USR command)
4. while CMD_REG != 0 { }         (poll until hardware clears)
5. Read W0â€“W3 registers           (copy data to destination buffer)
```

For reads larger than 16 bytes, the sequence repeats with incremented addresses.

#### Clock Configuration

Before flash reads, `configureFlashSPI()` sets up:

1. **MSPI clock divider**: PCR_MSPI_CLK_CONF_REG â†’ divider = 6
2. **SPI0/SPI1 clock**: CLOCK_REG â†’ CLK_EQU_SYSCLK (80 MHz)
3. **Dummy cycle correction**: At 80 MHz, dummy cycles need a -2 correction from the base value (DIO base = 3, corrected = 1)
4. **Read mode**: `configureSpiReadMode()` sets USER/USER1/USER2 for DIO

#### Source Files

| File | Function | Description |
|------|----------|-------------|
| `FlashRead.swift` | `configureSpiReadMode()` | Set SPI1 registers for DIO reads |
| `FlashRead.swift` | `readFlash()` | Read flash data in 16-byte chunks |
| `FlashRead.swift` | `readFlashUInt32()` | Read a 32-bit value from flash |
| `FlashRead.swift` | `readFlashUInt8()` | Read a byte (with 4-byte aligned read) |
| `FlashConfig.swift` | `configureFlashSPI()` | Clock setup + dummy fix + read mode |

## App Image Generation

### ELF â†’ ESP Image Conversion

Convert the ELF file built with Swift into an ESP32-C6 image using `Tools/elf2image.swift` (pure Swift replacement for esptool.py's `elf2image`).

### App Image Header Structure

ESP32-C6 app images begin with the following header (24 bytes):

| Offset | Size | Content |
|--------|------|---------|
| 0x00 | 1 | Magic byte (`0xE9`) |
| 0x01 | 1 | Number of segments |
| 0x02 | 1 | Flash mode (DIO=`0x02`) |
| 0x03 | 1 | Flash size/frequency |
| 0x04 | 4 | Entry point address |
| 0x08 | 16 | Extended header (WP pin, SPI drive settings, etc.) |

`elf2image.swift` automatically generates this header (same format as esptool.py), so manual construction is unnecessary.

### Required ELF Segments

`elf2image.swift` reads the loadable segments from the ELF and classifies them based on the following address ranges (same logic as esptool.py):

| Address Range | Classification | Description |
|---------------|---------------|-------------|
| `0x40800000`â€“`0x40880000` | IRAM/DRAM | Code/data in RAM (copied from Flash at boot) |
| `0x42000000`â€“`0x42800000` | IROM/DROM | Read-only code/data via Flash MMU |

## Reference

- Bootloader entry point: in IRAM region (set by linker)
- App entry point: `0x40800250` (IRAM region, typical)
- Bootloader segments: 1 (all IRAM)
- App segments: 5 (IRAM/DRAM + Flash MMU mapping)
